# Testing ADX Telemetry

## Quick validation

Run one configured agent and verify ADX rows:

```powershell
.\tests\Test-SingleAgent.ps1 -Agent ana.rodriguez
```

Run all configured agents sequentially:

```powershell
.\tests\Test-FullRun.ps1
```

Run selected agents in parallel:

```powershell
.\tests\Test-FullRun.ps1 -Agents laura.gomez,priya.sharma -Parallel -ThrottleLimit 3
```

## Azure Automation Concurrent Job Quota

Azure Automation limits concurrent running jobs per Automation account per region. Microsoft documents these default limits:

| Subscription type | Concurrent running jobs |
| --- | ---: |
| Enterprise / CSP | 50 |
| Pay-as-you-go, Sponsored, MSDN, MPN, Azure Pass | 10 |
| Free trial, Azure for Students, Azure in Open | 5 |

The limit counts all jobs in the Automation account and region, including scheduled runs and other tests. If a full run receives HTTP `429` for concurrent jobs, lower the throttle or let the built-in retry wait for capacity:

```powershell
.\tests\Test-FullRun.ps1 -Parallel -ThrottleLimit 2 -StartRetrySeconds 90 -MaxStartRetries 20
```

There is no higher Azure Automation pricing tier that directly increases this concurrency. Request a quota increase from Azure Support under **Service and Subscription limits (Quotas) > Azure Automation**. Microsoft states quota change requests do not have additional billing impact by themselves; billing remains based on Automation job runtime minutes.

## Activity Explorer Actor Expectations

When the runbook applies sensitivity labels through Graph/SharePoint, Activity Explorer can show `SHAREPOINT\system` as the user. That confirms the service-side label operation, but it is not the same as a user opening Office and applying/saving a label.

For a real demo where the `User` field shows the simulated employee, use Activity Explorer events generated by:

- Office web/desktop sessions signed in as the agent.
- Microsoft Purview Information Protection client/file labeler on an endpoint.
- Endpoint DLP on an onboarded Windows 11 or Windows 365 device.
- Exchange/Outlook user actions for labeled email scenarios.

Use the Azure Automation runbook as the data seeding and background activity engine. Use an endpoint or Office session for the small number of high-fidelity user-attributed screenshots.

Run a faster job-only validation without waiting for ADX ingestion:

```powershell
.\tests\Test-FullRun.ps1 -Parallel -ThrottleLimit 5 -NoADXWait
```

## Add expansion users

If a requested test user is not in `config\agents.json`, the test will skip that user. Add the user as an expansion agent first:

```powershell
.\tools\Add-StorylineAgents.ps1 -Search miguel
```

When prompted, select the user and choose to reset/store the password. This updates `config\agents.json`, `config\Installation_definitions.json`, Key Vault secrets, and Automation variables.

For profile-based expansion:

```powershell
.\tools\Add-StorylineAgents.ps1 -AutoFromProfiles -ResetPassword -StoreInKeyVault -UpdateAutomationVariables
```

If you only changed runbook code or ADX configuration, refresh the published runbook and `AgentConfig` without rotating secrets:

```powershell
.\tools\Publish-RunbookOnly.ps1
```

## Common ADX symptom

If jobs complete but ADX counts are zero, inspect the test `Warnings` and `Comments` columns. A warning like `[ADX] Failed to initialize ingestion: 401 (Unauthorized)` usually means the ADX `clientId` does not match the secret stored in Key Vault or the app is missing the `Database Ingestor` role.
